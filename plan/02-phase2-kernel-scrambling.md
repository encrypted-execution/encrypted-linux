# Plan 02 — Phase 2: Kernel Scrambling

**Goal:** Per-build syscall renumbering plus per-build kernel-internal
calling-convention scrambling. Closes the static-binary gap from Phase 1.
External kernel modules cannot load. Defeats pre-canned kernel ROP
chains and code-injection payloads. Module ABI is tied to the seed via
existing `modversions` machinery.

**Exit criterion:** A static `hello` built with stock GCC calls
`write(1, "hi\n", 3)` and gets `-ENOSYS` because what stock-userland
thinks is syscall 1 is now mapped to a different op in the kernel's
syscall table. Conversely, `hello` built against this build's
`asm/unistd_seeded.h` runs fine. Out-of-tree modules built against
stock kernel headers are refused by `insmod` with the same error path
that currently rejects `modversions` mismatches.

Estimated effort beyond Phase 1: 4–8 focused weeks. The work is wider
(many more touchpoints in the kernel) but each touchpoint is smaller
than Phase 1's GCC backend surgery.

## Why this matters

Phase 1 fails to stop static binaries (`research/06` §3). The kernel's
syscall ABI is the last surface available to a stock-built binary
shipped onto the box. Phase 2 closes it. Phase 2 also defeats the
class of kernel-side exploitation that Phase 1 left untouched — the
attacker who has read-write primitive into the kernel and wants to
call `commit_creds(prepare_kernel_cred(NULL))` finds that
`commit_creds` doesn't expect args in the registers their ROP chain
loads.

## Milestones

### M1 — Generate `asm/unistd_seeded.h` (week 1)

A small generator in the kernel source tree consumes the canonical
syscall table (`arch/x86/entry/syscalls/syscall_64.tbl`) plus the seed
and emits two artifacts:

1. `arch/x86/include/generated/asm/unistd_seeded.h` — exposes
   `__NR_<name> = HMAC-SHA256(seed, name)[0..15] mod 1024`, with
   collision resolution by sequential probing.
2. `arch/x86/include/generated/asm/syscall_seeded_table.S` — the
   kernel-side dispatch table at the permuted indices.

Properties:
- Bijective per seed (every canonical syscall is reachable).
- Deterministic given the seed.
- 32-bit and 64-bit numbers permuted independently (matching upstream).
- ioctl, prctl, futex *sub-op* numbers untouched in Phase 2 — they're
  a v2 milestone; the per-syscall surface is already large.

**Exit:** `nm vmlinux | grep sys_call_table` and a small kernel-side
selftest both confirm the permutation; flipping a bit in the seed
permutes the table.

### M2 — Match in libc (week 1, parallel to M1)

Patch musl's `arch/x86_64/syscall_arch.h` and friends so the syscall
stubs use `__NR_*` from the seeded header rather than the canonical
one. Buildroot has both kernel and musl building from the same seed
(plan/01 M4 already established that channel).

Kernel headers exported to userspace (`include/uapi/asm/unistd.h`)
are replaced for the encrypted-linux build with the seeded version.

**Exit:** A scrambled-userland binary calling `write()` traps to the
correct (renumbered) entry; a stock-userland binary calling `write()`
gets `-ENOSYS`.

### M3 — Scramble the kernel-internal calling convention (weeks 2–4)

Same compiler hooks as Phase 1 plan/01 M1+M3 — argument-register
permutation and callee-saved set permutation. Build the kernel with
the scrambling GCC, with a *different* sub-seed than userland (so
kernel-side gadgets are not reusable as userland-side gadgets, and
vice versa, even by an attacker who recovered one of them).

Critical preservation:
- **The syscall entry path** in `arch/x86/entry/entry_64.S` must
  still receive arguments in canonical kernel syscall registers
  (RDI/RSI/RDX/R10/R8/R9 from the userland `syscall` instruction);
  the trampoline at the kernel boundary translates into the
  scrambled kernel calling convention before calling
  `do_syscall_64`. (Same shape as Phase 1's libc syscall stubs,
  inverted.)
- **All hand-written .S** in `arch/x86/` audited — entry stubs,
  KVM VMENTRY/VMEXIT, copy_to_user, context switch, crypto. ~50
  files. Each either rewritten or annotated to use the canonical
  convention (with `noinline` + an explicit ABI attribute so GCC
  doesn't scramble the C side either).
- **DWARF/ORC unwinder** — kernel uses ORC, not DWARF, for
  unwinding (`research/03` §7's reference table). ORC tables are
  generated by `objtool` from the assembly; objtool runs after
  compilation and reads the actual prologue. If prologues change,
  ORC tables follow automatically. Confirm with an early smoke
  test: panic in a deep call stack, kernel emits a sensible
  backtrace.

This is the riskiest milestone of the entire project. The kernel
is enormous, full of subtle ABI-pinning patterns (`container_of`,
`pt_regs`, `task_struct.thread.sp`, `__attribute__((naked))`,
`switch_to`). Plan: start with a minimal `defconfig`, smallest
kernel that boots in QEMU; expand outward.

**Exit:** Scrambled kernel boots in QEMU with a scrambled userland.
Selftests pass. `objtool` is happy.

### M4 — Fold the seed into `modversions` CRC (week 4)

`research/06` §4 — a one-line change to `scripts/genksyms` (or the
new `gendwarfksyms`): include the seed in the symbol-CRC formula.
Result: any external module built against stock kernel headers
computes a different CRC for every exported symbol it imports;
`load_module()` rejects it via the existing path that already
rejects ABI-incompatible modules.

This is the cleanest single change in Phase 2. Reuses infrastructure
shipping in the kernel since 1996.

**Exit:** A trivial external module that prints from `init_module`
builds fine, loads fine. The same module built against stock headers
is refused at `insmod` with the existing "no symbol version" / "bad
CRC" error.

### M5 — vDSO (week 5)

The vDSO (`linux-vdso.so.1`) is kernel-built, mapped into every
userspace process, exporting `__vdso_clock_gettime` and friends with
a canonical SysV-ABI signature. Userspace must keep using the vDSO
for performance (`clock_gettime` 50ns vs. 200ns via syscall).

Two paths:
1. **Wrap in libc.** Per `research/04` §6 — libc generates per-seed
   thunks that re-marshal arguments from scrambled user-ABI into
   canonical vDSO-ABI. Simple, costs a couple of `mov` instructions
   per vDSO call.
2. **Build the vDSO with the scrambling GCC.** Kernel build invokes
   GCC; same toolchain in scope. The vDSO must then be exported
   under the *user* sub-seed, not the kernel sub-seed.

Pick (1) for the PoC — fewer moving parts. (2) is cleaner long-term.

**Exit:** `clock_gettime()` works from scrambled userland. A stock
static binary's vDSO call no longer works (because its registers
don't match the scrambled-user ABI the libc-thunk expects), and
falls through to the syscall path which then `-ENOSYS`'s.

### M6 — eBPF (week 5)

eBPF programs are JIT-compiled by the kernel at load time and called
from C kernel code via standard calling convention. The eBPF JIT lives
in `kernel/bpf/` + `arch/x86/net/bpf_jit_comp.c` and emits machine
code targeting the *standard* SysV ABI.

For Phase 2 to be complete, the eBPF JIT must emit code targeting the
*scrambled kernel* ABI. This is bounded — eBPF JIT is a few thousand
lines, one file per arch — but real engineering. Same problem libffi
poses for userspace (`research/04` §7) but a fixed in-tree target, so
tractable.

**Defer to a v2.1** if Phase 2 PoC needs to ship sooner; document the
gap. eBPF being broken is a feature, not a bug, from a hardening
standpoint (no untrusted eBPF can be loaded against a system whose
internal ABI it doesn't understand) — but it also breaks legitimate
tracing/observability tools. Position carefully.

**Exit:** Either eBPF works against scrambled kernel, or eBPF is
explicitly compiled out of the PoC kernel config and the README notes
it.

### M7 — Phase 2 demo (week 6)

The demo is a video / asciicast:

1. Boot scrambled kernel + scrambled userland. Show `uname -a`, `hello`,
   `cat /proc/cpuinfo`.
2. Stage a stock static `hello` (compiled on the host's stock GCC).
   `./stock-hello` → `Bad system call` (kernel returned ENOSYS,
   `write()` failed, libc trapped). Caught loudly.
3. Stage a stock external kernel module that exports a `/proc` entry.
   `insmod stock.ko` → `module: version magic '5.x.y …' should be …`
   (existing kernel error path, fired by our seeded CRC).
4. The scrambled `hello` and a scrambled external module work fine.

## What Phase 2 still doesn't do

- **vDSO under sub-seed without thunk.** M5 picks the wrapper path.
- **Per-syscall sub-op renumbering** (ioctl, prctl, futex op). Defer.
- **Kernel module distribution across multiple builds.** Each seed gets
  its own incompatible module ABI. This is desired.
- **eBPF if M6 deferred.**
- **CFI/IBT interaction audit.** kCFI lands fine on top of scrambled
  ABI because it operates at indirect call sites and uses its own
  type-hash mechanism (`research/06` §4). But the interaction wants a
  pass before shipping.
- **Hibernation, kexec, livepatch.** All consume kernel internals;
  each needs a per-feature audit.

## Stretch: ioctl renumbering (post-Phase-2)

The same trick as syscall numbers, applied to per-driver ioctl tables.
Most ioctls already use `_IO()`/`_IOR()`/`_IOW()` macros which embed
type+nr — extend the macros to consume the seed. Userspace headers
must be regenerated under the same seed. Out-of-scope for Phase 2.

## Defended threat profile after Phase 2

Combined with Phase 1, the system rejects:

| Threat | Phase 1 | Phase 2 |
|---|---|---|
| LD_PRELOAD malicious .so | ✓ (undefined symbol) | ✓ |
| Dynamic stock ELF dropped on box | ✓ (undefined symbol) | ✓ |
| Static stock ELF dropped on box | ✗ (syscalls work) | ✓ (`-ENOSYS`) |
| External kernel module | ✓ (modversions CRC) | ✓ (seed-folded CRC) |
| Pre-canned userspace ROP | partial (gadgets re-meaning) | partial |
| Pre-canned kernel ROP | n/a | partial (gadgets re-meaning) |
| eBPF code injection | n/a | ✓ if M6 done, ✗ if deferred |
| Logic bugs in scrambled apps | ✗ | ✗ |
| Data-only attacks | ✗ | ✗ |
| Attacker has compiler | ✗ | ✗ |
